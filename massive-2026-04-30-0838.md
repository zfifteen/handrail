## Synthesis

Strongest supported conclusion: the posted plan is recognizably aligned with incremental-coder, but Phase One is not strict enough. Extracting the existing phone root into `PhoneRootView` changes live behavior and should move to Phase Two, where the root coordinator is intentionally implemented. Phase One should add only new scaffold files, placeholder views, pending tests, deterministic fixtures, and the iOS test target.

The next most consistent finding is that validation must become more concrete inside each iteration. The plan names repo-native commands, which is good, but Phase Three should repeat the exact command block after every bounded section and include an iPhone compact-path build/test whenever shared root, transcript, store, command, or window behavior changes. Simulator destinations should be discovered or documented at startup instead of hard-coding names that may not exist locally.

The multiwindow state contract needs to move earlier. Before adding selected-chat windows, the plan should verify that `HandrailStore` does not encode one global "current chat" or one global "viewing chat" assumption that would make multiple windows fight over selection, lifecycle, or command availability. This is a state-model iteration, not just a UI iteration, and should happen before the multiwindow scene work.

Several Phase Three units are too broad for the stated process. The command iteration should split menu commands, keyboard shortcuts, and pointer affordances unless the implementation proves they touch the same narrow code. Transcript work should split parser extraction from renderer extraction. The final "previews and product smoke verification" item should be final acceptance or release verification, not a Phase Three implementation iteration with a feature commit.

The models agreed that deterministic fixtures should be part of Phase One. The plan currently mentions previews and pending tests, but it should explicitly introduce fixture data for paired/offline states, running chats, approvals, pinned chats, failures, completed chats, transcript blocks, and empty states. These fixtures become the invariant test substrate for later phases.

Lower-confidence findings came from models that could not see the linked comment. Their repo/spec-level concerns are still useful where they overlap with direct reviewers, but they should not be treated as direct evidence that the posted plan omitted something unless confirmed against the comment text.

## Concrete changes to make before implementation

1. Move `PhoneRootView` extraction from Phase One to Phase Two.
2. Add Phase One fixture files or builders for paired state, chats, approvals, pinned ordering, transcripts, offline state, and empty state.
3. Add explicit per-iteration validation blocks under every Phase Three item.
4. Replace brittle simulator destination names with a startup-discovered destination note, or state the required local simulator explicitly before work begins.
5. Split transcript extraction into parser extraction and renderer/view extraction.
6. Split menu, keyboard, and pointer work unless a repo inspection proves one narrow implementation point covers all three.
7. Add an early bounded iteration for shared selection/window state semantics before selected-chat windows.
8. Reframe Iteration 13 as final acceptance verification, not a normal Phase Three implementation section.
9. Add iPhone compact regression validation to every iteration touching `RootView`, transcript rendering, commands, store selection, or scene/window behavior.
10. Keep the blocker rule explicit: if existing CLI/state fields cannot represent an issue requirement, stop and document the missing contract rather than adding protocol fields opportunistically.
